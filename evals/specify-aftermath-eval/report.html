Findings

GAP-01Gap Phase 0b compile-gate skipped — discovered the blocker the hard way

The skill's Phase 0 has two steps. Step 1 (prover_capabilities) ran twice; step 2 (move_diagnostics scope: 'file') ran zero times. As a result, the agent only learned the package can't compile after running the full Phase 1 inventory and trying sui move build manually via Bash.

Impact. 7+ extra Bash calls, ~3 minutes of wasted budget, and a wasted full-package find/grep sweep. On a 229-function repo, deferring the compile-health gate is the difference between a 2-minute "blocked" report and a 15-minute one.

Recommendation. In skills/specify/SKILL.md §"Phase 0", elevate the move_diagnostics call from a step in prose to a hard gate: if it returns any severity: 'error' diagnostic mentioning unresolved external module or git fetch failures, surface them verbatim in .specify-progress.json as blocker.kind: "compile_unresolvable" and exit before Phase 1. Optionally add a sentence: "do not begin function inventory until the diagnostics pass returns ≥ one resolvable file".

GAP-02Gap Scope-narrowing is an undocumented heuristic

The skill spec treats "every externally-reachable function" as the target set and says Phase 2 asks the user per-function. The agent here improvised a three-level scope cut (package → file → 2 functions) entirely on its own. This is reasonable for an AMM with 229 surface functions and a "no-pause" caller, but it isn't in the skill.

Impact. Behavior is implementation-defined when scope is large. Different model checkpoints will narrow differently.

Recommendation. Add an explicit "Phase 1.5 — Scope" section to the skill: if external-fn count exceeds N (suggest 25), prompt the user with three choices (whole package, prioritized subset, single function); under non-interactive mode, pick "prioritized subset" using a documented ranking (e.g., invariant > swap > deposit/withdraw > getters > admin caps). Codifying it makes evals reproducible.

GAP-03Gap Verbal "no-pause" silently activates non-interactive contract

The skill's "Non-interactive (eval) mode" contract activates only when SPECIFY_AUTO_DEFAULTS=1 is in the environment. In this session the user said in chat "work without stopping for clarifying questions" and the agent applied the non-interactive contract anyway — skipping every AskUserQuestion gate, picking the first-option default, etc.

Impact. The right thing happened by accident: skipping AskUser was the only sensible response. But the contract is now bypassed via two unrelated channels (env var + verbal directive), with no recorded signal in the transcript that the agent took the non-interactive branch.

Recommendation. Either (a) extend the skill's non-interactive section to recognize a verbal directive ("if the user has explicitly said don't pause / proceed autonomously / no questions, behave as if SPECIFY_AUTO_DEFAULTS=1"), or (b) require the agent to surface a one-line acknowledgement in chat — "entering non-interactive mode: picking defaults at every gate" — so the elision is auditable.

BUG-01Bug Cross-tool flag confusion: --skip-fetch-latest-git-deps

The prover_capabilities response advertises --skip-fetch-latest-git-deps as a supported sui-prover 1.5.3 flag. The agent then attempted sui move build --skip-fetch-latest-git-deps — which fails on sui 1.69.2:

error: unexpected argument '--skip-fetch-latest-git-deps' found
Usage: sui move build [OPTIONS]

The flag exists on sui-prover the binary, but not on the sui move build subcommand of the Sui CLI. This is a category error: prover_capabilities exposes a flag list whose audience is the prover, not the Sui CLI, but a downstream agent reading the JSON cannot tell.

Impact. Low — one wasted Bash call. But the wrong inference is easy to make and survives many model refreshes.

Recommendation. In sui-prover-mcp's prover_capabilities response, label the supported_flags field explicitly as "sui_prover_flags" rather than the ambiguous "supported_flags". Optionally include a separate "sui_cli_flags" field probing sui move build --help. Alternatively, the skill text can pre-empt this by saying "supported_flags describes sui-prover's CLI, not sui move build's — never pass these to sui move build".

UX-01UX Probe greens-up on a fundamentally un-buildable package

Every Move.toml in this repo passed prover_capabilities's setup_warnings check (which inspects edition + explicit Sui/MoveStdlib deps). But the packages are not actually buildable on this machine. The probe's success/failure signal misled the agent into starting Phase 1 before any compile attempt.

Recommendation (server-side). Add a lightweight dep-reachability check to prover_capabilities: parse [dependencies] from the supplied Move.toml, for each git URL run git ls-remote --exit-code with a 3s timeout, and surface unreachable ones as setup_warnings: [{ kind: "private_dependency", url: "…" }]. Cheaper than a full sui move build --fetch-deps-only and catches the most common real-world blocker.

Recommendation (skill-side). Even without the server change, the skill's Phase 0b move_diagnostics step (see GAP-01) would catch this — so they reinforce each other.

GAP-04Gap prove_package dumps 105 KB of raw stdout on trivial build failures

The agent invoked prove_package on amm-math with no specs authored, partly as an empirical probe. The Move build failed before any spec evaluation could happen (named-address conflict, see GAP-05), yet the MCP tool returned the full sui-prover JSON envelope plus the entire prover raw_stdout — 107,265 characters / 105 KB — which exceeded the tool result token cap and got spilled to a sidecar file.

Impact. Forces a sidecar file read and Bash/jq parsing on what should be a one-line "build failed before specs" outcome. Costs an extra round-trip for every routine compile failure during /specify iteration.

Recommendation. In sui-prover-mcp's prove_package handler, detect the (exit_code ≠ 0 AND no spec evaluated AND raw_stdout contains compile-error patterns) shape and short-circuit: return a compact {kind: "compile_failure", reason: "", raw_stdout_path: ""} envelope under, say, 1 KB. Leave the sidecar for diagnostics but don't put it in the primary tool result. The same rule applies to dep-fetch failures and address-resolution failures.

GAP-05Gap Dep-substitution address conflicts surface as kind: "unknown"

When the agent applied Option B (local utilities clone), the patched Move.toml introduced a named-address conflict (utilities = 0x73baa782c5… from the local copy vs 0x10 from somewhere else in the dep graph). The MCP wrapper categorized this as kind: "unknown":

"findings": [{
  "kind": "unknown",
  "severity": "error",
  "message": "sui-prover exited with code 1 but produced no parseable findings"
}]

Impact. Without a structured kind, the skill's failure-taxonomy (references/failure-taxonomy.md) has no entry to match against. The agent must parse raw_stdout to discover what went wrong — exactly the pattern the taxonomy exists to prevent.

Recommendation. Extend the prover-output parser with finding kinds for the most common pre-spec failure modes:

  • kind: "dep_address_conflict" — when raw_stdout contains "Conflicting assignments for address"
  • kind: "unresolved_named_address" — when it contains "Unable to resolve named address"
  • kind: "unresolved_module" — "Unbound module"
  • kind: "dep_fetch_failure" — "Repository not found" or git-clone errors
  • kind: "no_specs_to_prove" — exit_code 0, zero specs found (currently silently invisible)

Then update references/failure-taxonomy.md with one row per kind, including the recommended fix (e.g., "address conflict → check duplicate [addresses] blocks in Move.toml, or run with --default-move-edition 2024").

BUG-02Bug summary reports 0/0/0/0 while a separate findings[] says "error"

The prove_package response on the address-conflict failure:

"summary": { "verified": 0, "failed": 0, "skipped": 0, "timeouts": 0 },
"findings": [{ "kind": "unknown", "severity": "error",
                "message": "sui-prover exited with code 1 but produced no parseable findings" }]

An agent (or a downstream eval rig) consuming only summary would conclude "nothing happened, all 0s" — which is true at the spec level but misleading: the run materially failed. The two fields disagree on whether anything went wrong.

Recommendation. Make summary include a non-numeric overall status — e.g., summary.overall: "compile_failure" | "no_specs" | "verified_all" | "failed_some" | "timeout". Numeric tallies stay for spec-level reporting, but the status field gives a single trustworthy answer to "did the run succeed?".

WIN-02Win Agent caught the stub-callee trap before drafting specs

After Option B unblocked the build, the agent read utilities::fixed source and noticed every function body is abort 0 (the public on-chain bytecode interface with private real source). It paused and said: "Wait — the public utilities is stub-only (abort 0 bodies for every fn). Let me see what sui-prover does with this before committing to a strategy." — then ran prove_package empirically rather than authoring specs that would silently verify against trivial abstractions.

Significance. This is the kind of awareness the skill spec doesn't currently teach. With stub callees, every caller's verification is vacuous (every path aborts), so a "verified" spec means nothing. The agent caught it; the skill should codify the check.

Recommendation. Add a Phase 1.5 callee-quality probe to the skill: for each callee inside an external function body, check the callee's module source — if every public function body is abort 0 or native, flag the dep as a stub and either (a) require user-supplied abstractions (#[spec_only] axiom) before specs can be drafted, or (b) downgrade the verification claim to "spec drafted, not verified — callee module is stub-only".

GAP-06Gap #[spec(target = ...)] body-call requirement is undocumented

The sui-prover's bytecode-transformation phase rejects a #[spec(target = fixed::mul_up)] function whose body doesn't call fixed::mul_up. The agent's first axiom drafts used fresh() bodies (a common Move Prover idiom) and got:

error: Spec function `specify_axioms::mul_up_spec` should call target function `fixed::mul_up`
   ┌─ ./sources/specify_axioms.move:33:5

The skill spec doesn't mention this constraint. Neither does skills/specify/references/failure-taxonomy.md. The agent recovered by reading asymptotic-code/sui-prover's pure-reachable.move example and switching to #[spec(skip, target = ...)] — but the path from error → fix required two iterations and a doc dive.

Recommendation. Add to references/failure-taxonomy.md:

spec_target_body_no_call:
  signal: "Spec function ... should call target function"
  cause: "#[spec(target = X)] requires the body to call X, or use the skip attr."
  fix:   "Add `skip` attribute: `#[spec(skip, target = X)]` — marks as opaque
          axiom without verifying the body. Use this for stub callees."
  example: pure-reachable.move

Also document the #[spec(skip, target = ...)] idiom as the canonical way to axiomatize external/stubbed dependencies — the skill currently doesn't teach this pattern at all.

UX-03UX raw_stderr ships full ANSI escape sequences

On a failed run, the MCP returned raw_stderr containing the prover's full colored Boogie error output — including ANSI escape sequences like [0m[1m[38;5;9merror[0m for every error line. A 5-error response was ~3 KB of escape codes wrapped around ~600 chars of actual content.

Impact. Agents and downstream tooling must strip ANSI codes before regex-matching the structured error text inside. Visible in the transcript, the agent's grep patterns had to account for the embedded escape sequences.

Recommendation. In sui-prover-mcp, strip ANSI escape codes from raw_stdout and raw_stderr before returning. Add a separate raw_stderr_ansi field if a UI ever wants the colored version, but the default should be machine-friendly plain text.

              ⏭️ specify_axioms::ln_spec at ./sources/specify_axioms.move:37
              ⏭️ specify_axioms::mul_down_spec at ./sources/specify_axioms.move:27
              ⏭️ specify_axioms::mul_up_spec at ./sources/specify_axioms.move:32
              Verification successful"
exit_code: 0

The MCP envelope:

"summary": { "verified": 0, "failed": 0, "skipped": 0, "timeouts": 0 }
"findings": []

Four ✅ verifications + five ⏭️ skips + a literal "Verification successful" line, and the structured fields report zero of everything. Any agent or eval rig that trusts the summary will conclude the run did nothing — the exact opposite of the truth.

Impact (critical). The skill's automation, the eval harness, and any future CI integration all depend on summary and findings being authoritative. They're not. Right now, "did the prover succeed?" requires re-grepping raw_stdout — at which point the structured envelope is dead weight.

Recommendation. In sui-prover-mcp's prove_package handler, parse the prover's per-spec output lines:

  • ✅ <qualified_name> → summary.verified++ and append {kind: "verified", function_under_test, location} to findings.
  • ❌ <qualified_name> at <loc> → summary.failed++ and append {kind: "failed", function_under_test, location, counterexample}.
  • ⏭️ <qualified_name> at <loc> → summary.skipped++ and append {kind: "skipped"}.
  • ⏱ <qualified_name> (timeout marker if used) → summary.timeouts++.

This is the single highest-leverage fix in the sui-prover-mcp surface — every other finding in this report becomes secondary if the structured output is wrong.

The skill currently says cross-package #[spec(target = ...)] generation is "deferred." It works in practice and should be promoted from deferred to documented.
